# 雪峰 Agent 混合检索升级方案

> 版本：v1.0 · 2026-06-24  
> 目标：在保留 SQL 事实检索主路径的前提下，引入向量语义层 + GraphRAG 关系层，用 GPU 加速离线建库与摘要生成。

---

## 1. 设计原则

| 原则 | 在本方案中的体现 |
|------|------------------|
| 事实归代码 | 分数/位次/冲稳保分档 **永远走 SQL**，不用向量替代 |
| 理解归模型 | 口语意图、复杂对比、政策解读 **交给 LLM** |
| 语义归向量 | 专业扩展、校名消歧、知识 RAG、结果重排 |
| 关系归图 | 地域、院校标签、专业层级、政策约束 |
| 主路径简单 | 常见查询走 L1→L3→L7；仅复杂问题升级到 GraphRAG |
| 失败可升级 | 每层输出可日志；SQL 无结果再扩专业；仍无结果再联网 |

---

|----|------|----------|

## 3. 查询编排器（核心链路）

### 3.1 标准路径（80% 查询）

```
用户输入
  → [L1] 结构化抽参 {province, category, rank, score, majors, region_avoid, tags, intent}
  → [L2] 向量扩展专业名 + 校名标准名（仅当 majors 非空或模糊时）
  → [L3] SQL 冲稳保查询（省=精确匹配, category, rank 区间, major IN 扩展列表）
  → [L4] 图过滤（地域排斥、标签要求、专业黑名单子图）
  → [L5] 向量重排（候选校 Top-20，按与用户完整意图的相似度）
  → [L7] 组装上下文 + 单次 LLM 生成
```

### 3.2 复杂路径（20% 查询，触发 GraphRAG）

触发条件（任一满足）：

- 意图为 `compare`（A校 vs B校）、`policy`（填报规则）、`career`（就业/考研路径）
- 用户明确问「为什么」「怎么选」「有什么区别」
- SQL 结果为空且向量扩展后仍为空
- 需要跨 3 个以上实体类型推理

```
  → [L6a] GraphRAG Local Search：从提及实体出发，遍历 1-2 跳子图
  → [L6b] GraphRAG Global Search：检索相关社区摘要（预生成）
  → [L6c] 向量 RAG：检索方法论知识 chunk（张雪峰知识库）
  → [L6d] 联网搜索（Tavily，可选）
  → 融合后交给 LLM
```

### 3.3 意图路由表

| intent | 主检索 | 辅助 |
|--------|--------|------|
| `recommend` | SQL + 图过滤 + 重排 | 联网补学校环境 |
| `compare` | 图子图 + GraphRAG | SQL 取两校分数 |
| `policy` | GraphRAG Global + 政策节点 | - |
| `major_info` | 向量 RAG + 图（专业→门类） | 联网 |
| `chat` | 仅 LLM | - |

## 4. 知识图谱设计（graph.db）

### 4.1 实体类型（nodes）

```sql
CREATE TABLE nodes (
  id          TEXT PRIMARY KEY,   -- 如 school:浙江大学
  type        TEXT NOT NULL,      -- province|city|school|major|discipline|tag|policy|batch
  name        TEXT NOT NULL,
  props       TEXT                -- JSON 扩展属性
);
```

| type | 示例 | props |
|------|------|-------|
| `province` | 浙江 | `{mode:"专业+院校", quota:80}` |
| `city` | 杭州 | `{region:"华东"}` |
| `school` | 浙江大学 | `{level:"985", city:"杭州"}` |
| `major` | 计算机科学与技术 | `{discipline:"工学"}` |
| `discipline` | 工学 | - |
| `tag` | C9 | - |
| `policy` | 浙江志愿填报规则 | `{text:"..."}` |
| `region` | 东北 | `{provinces:["黑龙江","吉林","辽宁"]}` |

### 4.2 关系类型（edges）

```sql
CREATE TABLE edges (
  src     TEXT NOT NULL,
  dst     TEXT NOT NULL,
  rel     TEXT NOT NULL,   -- 关系名
  weight  REAL DEFAULT 1,
  PRIMARY KEY (src, dst, rel)
);
```

| rel | 示例 | 用途 |
|-----|------|------|
| `LOCATED_IN` | 浙江大学 → 杭州 → 浙江 | 地域过滤 |
| `OFFERS` | 浙江大学 → 计算机科学与技术 | 学校-专业 |
| `BELONGS_TO` | 计算机科学与技术 → 工学 | 专业层级 |
| `HAS_TAG` | 浙江大学 → C9 | 标签筛选 |
| `IN_REGION` | 黑龙江 → 东北 | 区域排斥 |
| `SAME_TIER` | 浙江大学 ↔ 复旦大学 | 同层次对比 |
| `EMPLOYS_TO` | 计算机科学与技术 → 互联网 | 就业（知识层） |
| `GOVERNED_BY` | 浙江 → 浙江志愿填报规则 | 政策 |

### 4.3 建图数据来源（按优先级）

1. **从 admission 表自动抽取**（零成本，第一批）
   - `school` 节点、`major` 节点、`OFFERS` 边
   - `province` 节点、`LOCATED_IN`（需学校→城市映射表）
2. **手工种子表** `data/seeds/`（第二批）
   - `school_tags.csv`：C9、985、211、两电一邮、五院四系…
   - `school_city.csv`：学校→城市
   - `major_discipline.csv`：专业→学科门类（教育部目录）
   - `region_map.csv`：省份→区域
   - `province_policy.json`：各省志愿模式与数量
3. **LLM 辅助抽取**（第三批，GPU 离线跑）
   - 对缺失城市、标签的学校，用本地 LLM + 联网结果补全
   - 输出必须写回 `seeds/` 人工抽检后可复用

### 4.4 GraphRAG 社区与摘要

采用 Microsoft GraphRAG 思路，适配本地 GPU：

```
Step 1: 对图中每个连通社区做 Leiden/Louvain 社区发现
Step 2: 每个社区 → 收集节点+边+关联文本 → 送本地 LLM 生成摘要
Step 3: 摘要向量化存入 vector_communities
```

**预生成社区示例：**

| 社区 | 包含 | 摘要用途 |
|------|------|----------|
| C9 计算机 | 9校 + 计算机相关专业 | 「C9 计算机对比」类问题 |
| 江浙沪综合强校 | 复旦、上交、浙大、南大… | 地域偏好推荐 |
| 电气强校 | 华北电力、西电、武大电气… | 行业特色校对比 |
| 各省政策 | 省份 + policy 节点 | Global Search 政策问答 |

```sql
CREATE TABLE communities (
  id          TEXT PRIMARY KEY,
  level       INTEGER,        -- 0=细社区, 1=上级社区
  node_ids    TEXT,           -- JSON 数组
  summary     TEXT,           -- LLM 生成的社区摘要
  embedding_id TEXT           -- 指向 vector 库
);
```

## 5. 向量索引设计（vector/）

### 5.1 四套向量集合（LanceDB 推荐，本地文件、支持 GPU 批量写入）

| collection | 文档来源 | 用途 |
|------------|----------|------|
| `majors` | admission 表 major_name 去重 + 教育部专业目录描述 | 专业意图扩展 |
| `schools` | 校名 + 别名 + 城市 + 标签拼接 | 校名消歧、语义匹配 |
| `knowledge` | 张雪峰知识 chunk（专著/课程笔记） | 方法论 RAG |
| `communities` | GraphRAG 社区摘要 | Global Search |

### 5.2 Embedding 模型（GPU 推理）

**推荐：BGE-M3**（`BAAI/bge-m3`）

- 中文效果好，支持 dense + sparse（可选）
- `sentence-transformers` + CUDA 批量 encode
- 26 万条投档线不必全向量化——只对 **去重后的 major（约 5k）+ school（约 2k）+ 知识 chunk（约 2k）** 建索引，总量 < 1 万条

### 5.3 专业扩展流程（L2）

```python
def expand_majors(user_majors: list[str], top_k: int = 15) -> list[str]:
  q_emb = embed("想学" + "、".join(user_majors))
  hits = majors_collection.search(q_emb, top_k=top_k, threshold=0.55)
  return dedupe(user_majors + [h.name for h in hits])
```

输出喂给 SQL，**不直接用向量距离做冲稳保**。

## 7. GPU 使用方案

> 说明：本场景 **不需要训练大模型**。GPU 用于 **离线建库加速** 和 **本地 LLM 摘要生成**。

> **目标硬件：RTX 3070 Ti · 8 GB 显存**（已确认）

### 7.1 离线 Pipeline（`scripts/build_index.py`，一次性 / 数据更新时重跑）

```
┌─────────────┐    ┌──────────────┐    ┌───────────────┐
│ 1. 建图     │ →  │ 2. 向量编码   │ →  │ 3. 社区发现    │
│ graph.db    │    │ BGE-M3 GPU   │    │ Leiden (CPU)  │
│ CPU         │    │ ~2 GB 显存   │    │               │
└─────────────┘    └──────────────┘    └───────┬───────┘
                                               │
                    卸载 embedding 模型后       ▼
                                      ┌───────────────┐
                                      │ 4. 社区摘要    │
                                      │ Ollama GPU    │
                                      │ Qwen2.5-7B Q4 │
                                      │ ~5 GB 显存    │
                                      └───────────────┘
```

**8GB 铁律：同一时刻只跑一个 GPU 重任务。** 建库脚本分步执行，每步结束 `del model; torch.cuda.empty_cache()`。

### 7.2 推荐 GPU 工具链（3070 Ti 8GB 适配版）

| 任务 | 推荐模型 | 显存占用 | 8GB 是否可行 |
|------|----------|----------|--------------|
| Embedding 编码 | `BAAI/bge-m3` | ~2 GB（batch=32） | ✅ 宽裕 |
| 交叉编码重排 | `BAAI/bge-reranker-base` | ~1.5 GB | ✅ 在线可用 CPU 替代 |
| 社区摘要（离线） | `qwen2.5:7b-instruct-q4_K_M`（Ollama） | ~4.5–5 GB | ✅ 推荐 |
| 社区摘要（备选） | `qwen2.5:3b-instruct-q8_0` | ~3 GB | ✅ 更稳、略逊 |
| 7B 全精度 / FP16 | — | ~14 GB | ❌ 放不下 |
| Embedding 微调 | bge-m3 + LoRA | ~6 GB | ⚠️ 可做，第一版跳过 |

### 7.3 3070 Ti 8GB 推荐配置（`config/retrieval.yaml`）

```yaml
gpu:
  device: "cuda:0"
  vram_gb: 8

embedding:
  model: "BAAI/bge-m3"
  batch_size: 32          # 显存紧时改 16
  device: "cuda"

reranker:
  model: "BAAI/bge-reranker-base"
  device: "cpu"           # 在线默认 CPU；离线可选 cuda
  max_candidates: 80

local_llm:
  provider: "ollama"
  model: "qwen2.5:7b-instruct-q4_K_M"   # ollama pull 后使用
  # fallback: "qwen2.5:3b-instruct-q8_0"
  num_ctx: 4096
  temperature: 0.3

build_pipeline:
  sequential: true        # 强制分步，禁止并行占显存
  summary_batch: 1        # 社区摘要逐条生成
```

### 7.4 Ollama 安装与拉取（Windows）

```powershell
# 安装 Ollama 后执行
ollama pull qwen2.5:7b-instruct-q4_K_M

# 验证显存（任务管理器应显示 ~5GB 占用）
ollama run qwen2.5:7b-instruct-q4_K_M "用三句话总结C9高校计算机专业特点"
```

### 7.5 预估耗时（RTX 3070 Ti 8GB）

| 步骤 | 数据量 | 耗时 |
|------|--------|------|
| 从 admission 建图 | 26 万行 | ~2 min |
| BGE-M3 编码 | ~1 万条 | ~4 min（batch=32） |
| 社区发现 | ~200 社区 | ~1 min（CPU） |
| 社区摘要（7B Q4） | ~200 条 | ~40–50 min |
| **合计首次建库** | | **~50 min** |

在线查询时：

- **对话回答**：继续用 DeepSeek API（不占本地显存）
- **向量检索**：LanceDB 纯 CPU
- **重排**：默认 CPU；若需 GPU 加速，确保 Ollama 未运行

### 7.6 显存冲突规避清单

| 场景 | 做法 |
|------|------|
| 跑 `build_vectors.py` | 先 `ollama stop` 或退出 Ollama |
| 跑社区摘要 | 确保 Python 未加载 torch embedding 模型 |
| 本地调试 reranker GPU 版 | 单次最多 80 候选，batch=8 |
| 同时开游戏/其他 GPU 应用 | 改用小模型 `qwen2.5:3b` 或全 CPU |

### 7.7 可选进阶：Embedding 微调

仅当有 **真实用户 query → 专业映射** 标注数据（>500 条）时考虑：

- 基座：BGE-M3
- 方法：对比学习（in-batch negatives）
- 用途：提升「搞 AI」→「智能科学与技术」类口语映射

**第一版不做**，避免为过拟合评测而训练。
